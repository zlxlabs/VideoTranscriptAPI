## LLM 并发升级 & 转录处理流程全览

> 记录 2025-11-17 合并的 LLM 并发优化、结构化/非结构化处理路径，以及 CapsWriter / FunASR 的执行顺序，便于后续排障与迭代。

### 1. 本次改动摘要

- **LLM 并发调度**
  - `llm_executor`：在 `api/context.py` 中新增 LLM 专用线程池（默认 `concurrent.llm_max_workers=10`），`process_llm_queue` 出队后即提交到线程池；不同任务可并行执行，单任务由 `task_lock` 保证顺序。
  - `_handle_llm_task`：将 LLM 逻辑拆到独立 worker，结束后 `task_done()`；新增日志帮助定位“出队/开始/完成”。
- **短文本（CapsWriter / 平台字幕）**
  - `_process_original_logic`：校对与总结两个线程同步启动；长度不足 `min_summary_threshold` 时跳过总结。
- **长文本（非结构化）**
  - `_process_txt_segmented` / `_process_json_segmented`：
    - 说话人映射（若有）先执行。
    - `ThreadPoolExecutor(max_workers=10)` 同时运行分段校对与全文总结；总结直接使用原始 transcript + 映射，避免等待校对完成。
    - `SegmentedLLMProcessor.calibrate_text_segmented` 内部的段级 LLM 调用也读取 `llm.segmentation.concurrent_workers`（默认 10），因此单任务最多 10 段并发。
- **FunASR JSON 特殊处理**
  - 若未触发结构化逻辑，仍属于非结构化路径。现在在提交校对/总结前会基于 FunASR JSON + speaker mapping 构造 `summary_source_text`，确保总结与校对并行。
- **结构化模式（FunASR + 说话人识别 + platform/media_id）**
  - 仍由 `process_llm_task_with_structure` 执行：说话人推断 → 结构化校对（内部 chunk 并发）→ 结构化总结。校对完成后才进入 `_get_or_generate_summary`，以保证结构化输出可用。
- **测试**
  - `tests/unit/test_llm_concurrency.py` 使用 barrier 验证两个任务可同时进入 LLM 处理，同时断言缓存/通知及 `task_done()` 调用。

### 2. 当前转录文本处理流水线

| 场景 | 判定条件 | 处理模块 | 并发策略 |
| --- | --- | --- | --- |
| 平台字幕 / CapsWriter 短文本 | `use_speaker_recognition=False` 且 `len(transcript) < min_summary_threshold` | `_process_original_logic` | 仅校对线程；总结跳过 |
| 平台字幕 / CapsWriter 中长文本 | `use_speaker_recognition=False`，长度超过阈值 | `_process_original_logic` 两线程 + `_process_txt_segmented`（分段） | 校对线程 & 总结线程并行；分段校对内部最多 10 个线程 |
| FunASR（未触发结构化） | `use_speaker_recognition=True` 但缺少 `platform/media_id` 或结构化失败 | `_process_json_segmented` | 先提取 speaker mapping，再同时启动分段校对 + 总结；段内并发同上 |
| FunASR + 结构化 | `use_speaker_recognition=True` 且具备 `platform/media_id/transcription_data` | `_process_with_structured_output` / `process_llm_task_with_structure` | 说话人推断 → 结构化校对（chunk 并发）→ 结构化总结（串行） |
| LLM 队列调度 | 所有 LLM 任务 | `_handle_llm_task` 由 `llm_executor` 承载 | 全局最多 `llm_max_workers` 个任务并行，单任务受 `task_lock` 限制 |

### 3. 日志定位提示

- **队列**：`LLM任务出队...提交到线程池`、`开始处理LLM任务`、`LLM任务处理完成`。
- **短/长文本**：`TXT/JSON长文本校对/总结任务开始/完成`；短文本 `校对任务开始/完成` 等。
- **结构化**：`执行说话人推断`、`结构化校对完成`、`生成新的总结`。
- 通过筛选 `task_id` 即可还原单个任务的执行顺序与并发情况。

### 4. 后续关注点

- 若要让结构化总结也与校对并行，需重构 `process_llm_task_with_structure` 使其在说话人推断后即可提供“总结输入”。
- LLM API 并发大幅提升后，需注意模型速率限制与 WeCom 通知限流，必要时在外层加 throttling。
