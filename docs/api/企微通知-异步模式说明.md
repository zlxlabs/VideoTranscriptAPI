# 企业微信通知 - 完全异步模式说明

## 📋 概述

从 2025-10-19 起，项目的企业微信通知系统已升级为**完全异步模式**，以提升系统吞吐量和响应速度。

---

## 🔄 模式对比

### **旧模式：同步等待**

```python
# 发送消息
result = notifier.send_markdown_v2(content)
# ⚠️ 阻塞等待最多 120 秒
result.wait(timeout=120)
# 确认发送成功后才返回
return result.is_success()
```

**问题**：
- ❌ 阻塞工作线程（最长 120 秒）
- ❌ 遇到频率限制时等待 65 秒
- ❌ 降低系统吞吐量
- ❌ 3 个工作线程可能全部被通知等待占用

---

### **新模式：完全异步**

```python
# 发送消息
result = notifier.send_markdown_v2(content)
# ✅ 立即返回，不等待
# wecom-notifier 在后台自动处理
return True  # 表示已提交，不表示已送达
```

**优势**：
- ✅ 不阻塞工作线程
- ✅ 高吞吐量
- ✅ 自动处理频率限制（后台等待 65 秒）
- ✅ 自动重试网络错误
- ✅ 自动分段发送超长文本

---

## 🎯 核心特性

### 1️⃣ **全局单例模式**

为确保正确的并发控制和消息顺序，系统在启动时初始化**全局唯一的 WeComNotifier 实例**：

```python
# server.py - 应用启动时
@app.on_event("startup")
async def startup_event():
    # 初始化全局 WeComNotifier 实例
    init_global_notifier()
```

**为什么需要全局单例？**

- ✅ 同一 webhook 的所有消息由**同一个 WebhookManager** 处理
- ✅ 确保消息**严格按顺序**发送
- ✅ 并发控制正确生效（避免多个实例导致限流失效）
- ✅ 避免资源浪费（不重复创建/销毁实例）

**对比：**

| 模式 | 实例创建 | 并发控制 | 消息顺序 | 资源效率 |
|------|---------|---------|---------|---------|
| **旧：每次创建** | 每次发送 | ❌ 失效 | ❌ 可能乱序 | ❌ 低 |
| **新：全局单例** | 应用启动时 | ✅ 正确 | ✅ 保证有序 | ✅ 高 |

### 2️⃣ **后台自动处理**

wecom-notifier 会在后台线程中自动处理：

| 功能 | 说明 |
|------|------|
| **频率限制** | 遇到 45009 错误时，自动等待 65 秒后重试（最多 5 次）|
| **网络错误** | 遇到网络问题时，指数退避重试（2s → 4s → 8s）|
| **文本分段** | 超过 4096 字节自动分段，每段间隔 ~2 秒发送 |
| **消息顺序** | 同一 webhook 的消息严格按顺序发送（全局单例保证） |

### 3️⃣ **不阻塞工作线程**

```
场景对比：

旧模式：
00:00 - 任务1发送通知，阻塞工作线程1（等待65秒）
00:05 - 任务2发送通知，阻塞工作线程2（等待65秒）
00:10 - 任务3发送通知，阻塞工作线程3（等待65秒）
00:15 - 任务4到达，无可用工作线程，排队等待！ ❌

新模式：
00:00 - 任务1发送通知，立即返回，继续处理 ✅
00:01 - 任务2发送通知，立即返回，继续处理 ✅
00:02 - 任务3发送通知，立即返回，继续处理 ✅
00:03 - 任务4发送通知，立即返回，继续处理 ✅
（通知在后台自动处理，不影响任务处理）
```

### 4️⃣ **高可靠性**

即使主程序崩溃，wecom-notifier 的后台线程仍会尝试完成消息发送。

## 🔍 返回值含义变化

### **旧模式**

```python
success = notifier.send_markdown_v2(content)
# success = True  表示：消息已成功送达
# success = False 表示：发送失败
```

### **新模式**

```python
success = notifier.send_markdown_v2(content)
# success = True  表示：消息已成功提交到发送队列
# success = False 表示：提交失败（webhook 未配置、内容为空等）
```

**注意**：
- ✅ `True` 不代表消息已送达，只代表已提交
- ✅ 实际发送状态需要查看 `wecom_notifier` 日志
- ✅ 业务逻辑**不应依赖**通知发送是否成功

## 📝 使用建议

### ✅ **推荐做法**

```python
# 1. 发送通知，不关心结果
notifier.notify_task_status(url, "转录完成", title=title)
# 继续执行业务逻辑

# 2. 发送长文本，不等待
send_long_text_wechat(title, url, text, is_summary=True)
# 立即返回，继续处理下一个任务
```

### ❌ **避免的做法**

```python
# ❌ 不要依赖返回值判断发送成功
if notifier.send_text(content):
    # 错误：这只表示提交成功，不表示送达
    save_send_status("sent")  # 不准确

# ❌ 不要在关键路径中等待通知
try:
    process_transcription()
except Exception as e:
    # ❌ 错误：可能阻塞异常处理
    if not notifier.send_text(f"异常: {e}"):
        raise  # 通知失败不应影响异常处理
```

### ✅ **正确的错误处理**

```python
# ✅ 通知和业务逻辑分离
try:
    result = process_transcription()
    # 发送成功通知（不等待结果）
    notifier.notify_task_status(url, "转录完成")
    return result
except Exception as e:
    # 发送失败通知（不等待结果）
    notifier.notify_task_status(url, "转录失败", str(e))
    raise  # 继续抛出异常，不受通知影响
```

## 🔧 调试和监控

### **查看发送状态**

由于是异步模式，需要通过日志查看实际发送状态：

```bash
# 查看 wecom_notifier 日志
grep "wecom_notifier" logs/api_server.log

# 关键日志示例：
INFO:wecom_notifier:Message sent successfully
INFO:wecom_notifier:Message xxx sent successfully (3 segments)
WARNING:wecom_notifier:Rate limit exceeded, waiting 65s...
ERROR:wecom_notifier:All retries failed. Last error: xxx
```

### **日志级别说明**

| 级别 | 说明 |
|------|------|
| `DEBUG` | 消息提交、队列状态 |
| `INFO` | 消息发送成功、分段信息 |
| `WARNING` | 频率限制、网络重试 |
| `ERROR` | 发送失败、超过重试次数 |

## 🎯 最佳实践总结

1. **业务优先**：通知失败不应影响业务逻辑
2. **异步思维**：发送后立即继续，不等待结果
3. **日志监控**：通过日志追踪实际发送状态
4. **容错设计**：即使通知失败，业务也能正常完成
