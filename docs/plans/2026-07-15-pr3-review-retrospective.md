## 一、过程复盘：出现过的问题

### 1. 本地悬空提交（差点丢工作）

整理分支时手滑把本地 `main` 重置到了远程 `main`，导致此前积累的 30 个未推送本地提交瞬间失去分支引用，处于随时可能被 Git 垃圾回收的悬空状态。靠 `git reflog` 找回了丢失的 commit SHA，重新挂到新分支上，没有真的丢东西，但这是一次真实的操作失误，值得记一笔提醒自己：**在对一个长期使用的本地分支做 `reset --hard` 之前，先确认这个分支没有还没推送的独有提交**。

### 2. 挑提交时发现重复内容

从悬空提交里筛选要放进 PR3 的内容时，发现有 4 个提交（说话人推断相关）已经在更早的 PR2 里合并进 main 了。用 `git diff` 对比确认后排除，只保留真正未合并过的 26 个提交。这是分支拆分工作的常规去重步骤，没有造成实际问题，但提醒：**从一条长期存在的本地分支里回收内容时，必须先核对哪些内容已经通过其他路径（squash-merge）合并过**，否则会把同样的代码重复提交。

### 3. Worktree 清理时的"幽灵改动"

PR3 合并后清理构建用的临时 worktree 时，`git status` 显示一堆 "M"（已修改未提交）的文件，一度怀疑是不是有工作没保存。核实后确认只是该 worktree 的文件系统状态和 Git 索引没同步好造成的技术性噪音（内容早就通过正常提交流程进了 main），可以放心删除。

### 4. 打地鼠式的同类漏洞（最费工夫的一块）

云端 CI 第一次跑 PR3 时 fail，发现 3 个问题，其中最严重的一个是：**审计接口（`/api/audit/history`、`/filter-options`、`/summary`）用截断后的 API key（只保留首尾各 4 位）做租户权限校验边界，两个不同用户的 key 只要掐头去尾后长得一样就会碰撞，导致租户 A 能看到租户 B 的历史记录**。

修复这个问题（改用真正唯一的 `user_id` 做边界）之后，本地复查又在 `/summary` 接口挖出一个更隐蔽的变种：**如果某个多用户配置项漏填了 `user_id` 字段，系统仍会正常签发登录 token，但权限校验代码里 `if task_id and user_id:` 这个判断会因为 `user_id` 为空直接短路跳过整段校验，等于对任何任务都放行**。修完这个，再检查 `/calls` 接口发现了一模一样的漏洞模式（`user_id` 缺失时会误触发一个专门设计给"管理员查看全部记录"用的代码分支）。

这三处本质上是同一个根源问题的三个变种：**代码在多个地方分别假设"认证通过的调用方一定有合法的 user_id"，但实际配置流程只对缺失字段发警告、不拒绝**。这类"同一个假设漏洞在多个相似代码路径里各埋一份"的模式，教训是：发现一处后应该主动系统性排查同类端点，而不是等审查一个个揪出来（这次我在第三处修复后确实做了一次主动排查，确认没有第四处遗漏）。

### 5. 我自己的一次误判

"截断 key 撞车导致跨用户泄露"这个问题，我最初评估触发条件苛刻（需要两个 key 首尾恰好相同）、影响面小，判断为可以先记成技术债、往后拖。结果云端 CI 直接判定它是必须当场修复的阻塞级问题——原因是它给出了更具体的论据：项目里 key 的生成规则本身就没有强制的高熵随机性保证，实际碰撞门槛比我想的低，而且审计表本来就已经有 `user_id` 字段可以直接换用，修复成本比我预估的低。这次误判被 CI 纠正，提醒自己：**评估安全问题能否延后时，不能只看"理论触发概率"，还要看"修复成本是否真的高"——如果修复成本低，即便触发概率低也应该当场处理，没有拖延的必要**。

### 分层缓存并发场景下 task_status 与 llm_status.json 短暂不同步

**What:** 同一媒体近乎同时收到两个不同 `processing_options` 的请求时（如一个只要校对、另一个只要总结），先完成的那个请求会把自己的任务行状态记录成"当时看到的真实状态"，但这个状态后续可能被另一个请求的完成结果"追认推翻"，而已经落库的任务行不会被回头修正。

**Why:** `task_status` 是"任务完成时的快照"，`llm_status.json` 是"媒体当前的共享状态"，两者含义本来就不同，只是容易被误认为应该总是一致。

**Pros:** 明确这个语义区别后，未来任何要展示"媒体当前状态"的功能（如审计 UI）都应该去读共享状态而不是某一次任务的快照，能提前避免误用。

**Cons:** 触发前提很窄（需要同一媒体近乎同时提交至少两个不同配置的请求），个人单用户场景概率低；真正修复涉及"任务行 vs 共享状态谁是真相源"的架构取舍，改动范围不小。

**Context:** 相关代码在 `src/video_transcript_api/api/services/llm_ops.py::_save_llm_results()`（media_lock 保护范围）和 `cache_manager.get_task_by_view_token()` 的任务行选取逻辑。

**Effort:** M（人工）→ M（CC）

**Priority:** P3

**Depends on:** 无

### 审计端点用截断 API key mask 做鉴权边界（已修复主漏洞，遗留配置校验缺口）

**What:** 本次已经把 `/api/audit/history`、`/filter-options`、`/summary`、`/calls` 全部改用 `user_id` 做租户边界，堵住了 mask 碰撞和 `user_id` 缺失绕过校验这两类真实漏洞。遗留的是：**系统本身不校验多用户配置里各个 `user_id` 是否互不重复**——如果运维手滑给两个不同租户填了相同的 `user_id`，这两个租户会共享同一个"身份"，重新在这几个端点之间互相可见。

**Why:** `user_id` 唯一性目前完全依赖运维手工保证，没有任何加载时校验兜底。

**Pros:** 加一道配置加载时的查重校验，成本不高，能把"配置错误"和"代码漏洞"两类风险都堵上。

**Cons:** 目前只有你自己在用，触发这个场景需要主动配置多用户且手滑重复，实际发生概率很低。

**Context:** 相关代码在 `src/video_transcript_api/utils/accounts/user_manager.py::_load_users_config()`，目前只对缺失必需字段打 warning、不拒绝配置项，也没有做 `user_id` 去重检查。

**Effort:** S（人工）→ S（CC）

**Priority:** P3

**Depends on:** 无

### `calibrate=False` + 说话人识别时，说话人姓名推断仍会调用 LLM（产品决策，非纯 bug）

**What:** 用户关闭校对（`processing_options.calibrate=False`）但同时启用了说话人识别时，"把 Speaker1/Speaker2 猜成真实姓名"这一步依然会跑，可能产生 LLM 调用和 token 成本——这与"只转录=零 LLM 成本"的直觉预期有落差。

**Why:** 这是实现 `skip_calibration` 功能时的有意设计（说话人姓名推断被归类为"转录交付物"而非"校对"），有专门的注释和测试锁定这个行为，不是疏忽。是否要改变这个行为，是一个产品取舍问题，不是技术对错问题。

**Pros:** 维持现状能让用户在关闭校对时依然拿到高质量的说话人姓名，这是有价值的功能。

**Cons:** 与文档"只转录不产生 LLM 成本"的字面预期有出入（文档已补充说明这个例外）；如果想要真正的"零成本、保留原始占位符"选项，需要新增一个独立开关。

**Context:** `src/video_transcript_api/llm/processors/speaker_aware_processor.py::process()`；文档说明见 `docs/features/processing_options.md`。三个方案：(A) 维持现状；(B) `calibrate=False` 时彻底跳过姓名推断；(C) 新增独立开关由用户自己选。

**Effort:** 视方案而定，(A) 零成本 / (B) S / (C) M

**Priority:** 待你拍板

**Depends on:** 无
